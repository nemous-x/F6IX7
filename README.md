# F67 — AI Development Operating System for Claude Code

F67 makes Claude behave like a senior engineer who has been on your project since day one. Instead of prompt → search → code → forget, every request runs through a deterministic pipeline backed by Domain Driven Memory that lives in your repository and grows with it.

```
Detect domains → Load memory → Discover context → Build context → Build spec
→ Plan → Decompose → Implement (one task) → Test → Review → Improve → Evolve memory
```

Understanding always comes first. The AI never starts by writing code.

## Install

In Claude Code, add the marketplace and install the plugin:

```
/plugin marketplace add nemous-x/nemous
/plugin install f67@nemous
```

Then in any project:

```
/f67-init
```

This analyzes your repository (stack, architecture, conventions, testing, UI, business domains) and generates the initial Domain Driven Memory and relationship graphs under `.claude/f67/`. Everything is local, plain files, inside your repo. No cloud, no database.

## The five pillars

1. **Domain Driven Memory** — knowledge organized by business domain (authentication, billing, orders…), not by folder. Each domain keeps overview, business rules, architecture, APIs, database, UI, tests, related files, patterns, known issues, decisions, and history. Memory never depends on chat history.
2. **Context first** — every workflow starts by gathering context: affected domains, existing features, relevant APIs, tests, rules. Planning and coding come after.
3. **Specialized agents** — twelve dedicated subagents, one per pipeline phase. The main session only orchestrates; it never implements, reviews, or reads large codebases.
4. **Prompt engineering** — user requests become structured execution specifications (`prompt-spec.md`) that drive every later phase.
5. **Progressive intelligence** — every completed workflow feeds back into memory. The project gets easier to work on over time.

## Commands

| Command | Purpose |
|---|---|
| `/f67-init` | Onboard a repository: analyze, detect domains, generate initial DDM and graphs |
| `/f67-discover` | Detect affected domains and collect relevant files for a topic |
| `/f67-memory` | Inspect, audit, or rebuild Domain Driven Memory |
| `/f67-prompt` | Transform a request into an execution specification |
| `/f67-brainstorm` | Approaches, tradeoffs, alternatives — no code |
| `/f67-explain` | Explain architecture, flows, dependencies — read-only |
| `/f67-plan` | Implementation plan, milestones, and decomposed task tree |
| `/f67-execute` | Fast path: short tasks end to end in one dispatch — context, implement, verify, memory update |
| `/f67-implement` | Execute one task from the plan (parallel dispatch for independent tasks) |
| `/f67-test` | Generate and run tests, edge cases, regression validation |
| `/f67-review` | Full code review with severity-ranked findings |
| `/f67-improve` | Prioritize and apply improvements from review |
| `/f67-docs` | Generate or update documentation from memory |
| `/f67-sync` | Synchronize memory and graphs with repository changes |

## Typical workflows

Short task — one command, seconds of overhead:

```
/f67-execute Rename the export button on the invoices page
```

Feature — full pipeline:

```
/f67-prompt Implement partial refunds
/f67-plan
/f67-implement        # repeat per task; independent tasks run in parallel
/f67-test
/f67-review
/f67-improve
/f67-sync             # reconcile memory with the repo
```

F67 is built to be fast: complexity-based routing (small work never pays full-pipeline overhead), parallel agent dispatch, hard token budgets on every agent output and artifact, model routing (fast models for detection/memory, powerful models for planning/implementation/review), and memory that updates continuously — every completed task writes its delta immediately.

Each stage writes an artifact (`prompt-spec.md`, `implementation-plan.md`, `execution-report.md`, `review-report.md`, `improvement-plan.md`) into `.claude/f67/artifacts/<NNN>-<slug>/`. Later stages consume artifacts, not conversation history — you can close the session between any two stages.

## What lives where

- **This plugin** — agents, command skills, templates, schemas, and skill-injection rules. Portable and stack-agnostic: F67 discovers and injects the skills installed in your environment, and requests missing ones based on your project's needs.
- **Your repo (`.claude/f67/`)** — memory, graphs, state, artifacts. Generated by `/f67-init`, evolved by `/f67-sync`. Commit it; it is your project's brain.

Curated knowledge is protected: files marked `<!-- f67:curated -->` are only ever appended to under a `## Learned` heading, never rewritten by agents.

## Repository layout (plugin)

```
.claude-plugin/plugin.json   Manifest
agents/                      12 pipeline agents
skills/                      13 /f67-* command skills
templates/                   Memory, artifact, and state templates + skill-injection rules
schemas/                     graph.schema.json
docs/                        f67-core.md (canonical conventions), architecture.md
```

See `docs/f67-core.md` for the full runtime layout, graph format, artifact contracts, and state discipline.
